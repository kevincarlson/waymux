# ADR-002: Frame Encoding Strategy

**Status:** Accepted  
**Date:** 2025-05-03  
**Deciders:** Kevin (project lead)

---

## Context

Every rendered frame from the compositor must be transmitted over the Unix domain socket to the Android client. An uncompressed 1080p BGRA8 frame is approximately 8.3 MB. At 60 fps this is ~500 MB/s — well within what a local Unix socket can handle on modern ARM hardware, but it imposes a significant CPU cost to copy and a significant memory pressure from in-flight buffers.

Candidate encoding strategies:
1. **Raw BGRA8** — uncompressed passthrough
2. **Zstd-compressed BGRA8** — lossless compression, software
3. **H.264 / AV1 via hardware encoder** — lossy or lossless, hardware-accelerated
4. **Damage-region-only updates** — send only changed pixels (orthogonal to encoding, combinable)

---

## Decision

Support a **tiered encoding pipeline**:
- **Tier 0: `RawBgra8`** — always available; used for development, testing, and fallback.
- **Tier 1: `ZstdBgra8`** — default for production; lossless, low-latency, CPU-only. Zstd level 3 by default (good balance of ratio and encode speed on ARM).
- **Tier 2: `H264AnnexB`** — reserved in the protocol (`encoding_id = 0x02`) but **not implemented in M1–M5**; deferred until hardware encoder access via Android `MediaCodec` is confirmed feasible from Termux.

Damage-region updates (sending only changed pixel regions) are implemented **alongside** tier 1/2 encoding when the compositor provides damage hints via `zwlr_screencopy_frame_v1::damage` events.

---

## Rationale

### Why Zstd as the default

- Zstd at level 3 achieves ~3–5× compression on typical desktop frame content (text-heavy UIs), reducing 8.3 MB/frame to ~2–3 MB/frame, cutting socket bandwidth by 60–70%.
- Encode latency at level 3 is approximately 3–8 ms on an ARM A-class core, comfortably within a 16.7 ms frame budget.
- The `zstd` Rust crate (`facebook/zstd`) is mature, well-maintained, and cross-compiles cleanly to `aarch64-linux-android`.
- Lossless: no visual artifacts. This is important for a desktop environment where text legibility is critical.

### Why not H.264 immediately

- H.264 hardware encoding via Android `MediaCodec` is not accessible from Termux processes because `MediaCodec` requires Android framework services unavailable to Termux's Linux userland.
- Software H.264 (x264) encoding at 1080p/60 is CPU-prohibitive on mobile ARM cores.
- H.264 introduces latency (B-frames, reference frame management). Zstd operates frame-independently, giving zero additional latency.
- H.264 is reserved in the protocol ID space so a future implementation (e.g., via a MediaCodec tunnel from the Android client side) can be added without breaking protocol compatibility.

### Damage regions

- When wlr-screencopy provides damage hints, encoding only the union of changed rectangles dramatically reduces payload size for typical desktop usage (cursor movement, text cursor blink, partial redraws).
- The `WfpMessage::FrameDamage` message type is defined in the protocol to carry per-region encoded data.
- Damage tracking requires maintaining a per-client "last sent" damage state in `waymux-bridge`, which adds modest memory overhead (one `Vec<DamageRect>` per session).

---

## Consequences

- The `FrameEncoding` enum in `waymux-proto` must be versioned carefully; future encoding IDs (H.264, AV1) must not conflict.
- The `WAYMUX_ENCODING` config variable allows operators to select `raw` for debugging (disables compression to expose protocol issues).
- The Bridge encoder is structured as a `FrameEncoder` trait so new encoding tiers can be plugged in without changing the pipeline architecture.
- Clients must advertise their supported encodings in a future handshake message (deferred to a post-M5 protocol version; v1 assumes the client supports whatever the bridge sends).
