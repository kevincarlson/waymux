# ADR-001: Unix Domain Socket as Transport

**Status:** Accepted  
**Date:** 2025-05-03  
**Deciders:** Kevin (project lead)

---

## Context

Waymux Client and Waymux Bridge must exchange high-bandwidth frame data (up to ~250 MB/s uncompressed for a 1080p/60 display) and low-latency input events. The two processes run on the same physical Android device: one inside Termux (Linux userland), the other as an Android application.

Candidate transports were:
1. **Unix domain socket** (local filesystem path)
2. **Android Binder IPC** (via AIDL service)
3. **TCP loopback** (`127.0.0.1` on a chosen port)
4. **Shared memory + eventfd signalling**

---

## Decision

Use a **Unix domain socket** at `$TMPDIR/waymux.sock` (Termux side), accessible from Android via the shared filesystem path under Termux's data directory.

---

## Rationale

| Criterion | Unix socket | Binder | TCP loopback | SHM + eventfd |
|---|---|---|---|---|
| Bandwidth | High (kernel zero-copy path) | Medium | Medium (loopback) | Highest |
| Latency | Very low | Low | Low | Lowest |
| Android ↔ Termux accessibility | Via sharedUserId / filesystem | Requires AIDL service registration | Works natively | Complex cross-process SHM setup |
| Implementation complexity | Low (standard POSIX) | High (requires system service) | Low | High |
| Requires root / special signing | No (sharedUserId or Termux API) | No (but complex) | No | Possibly yes |
| Precedent in ecosystem | Termux:X11 uses exactly this pattern | Unusual for Termux addons | Common but adds DNS overhead | Uncommon in this context |

The **Termux:X11** project demonstrates that a Unix socket in `$TMPDIR` shared between Termux and a companion Android app (using `sharedUserId`) is a well-understood, working pattern for exactly this use case. We follow that precedent rather than inventing a new approach.

Binder was rejected because it requires registering an Android service and writing AIDL definitions, adding significant complexity and coupling to the Android service lifecycle.

TCP loopback was rejected because it adds unnecessary protocol overhead (TCP headers, checksums, congestion control) for purely local communication, and it would require a port allocation scheme.

Shared memory was rejected as a primary transport because it is significantly harder to implement correctly cross-process on Android without root, particularly for dynamic-sized frame buffers. It may be revisited in future for a zero-copy optimization layer (ADR-002 notes this).

---

## Consequences

- The Waymux Client Android app must have the `com.termux.permission.RUN_COMMAND` permission, or be signed with the same key as the Termux app to use `sharedUserId`. Both options are documented in the client `SPEC.md`.
- The socket path is configurable via `WAYMUX_SOCKET` env var to support non-standard Termux installations.
- Frame buffer data will be copied from the shm Wayland buffer into the socket, introducing one copy per frame. This is acceptable given the alternative complexity. Future optimization (ADR-002) will use compression to reduce the byte volume.
- The protocol uses length-prefixed framing (4-byte LE length header) to delineate messages over the stream socket.
