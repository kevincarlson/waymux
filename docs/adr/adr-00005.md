# ADR-005: Widget Toolkit for Light Speed Desktop Chrome — Iced (preferred) with GTK4 fallback

**Status:** Accepted  
**Date:** 2025-05-03  
**Deciders:** Kevin (project lead)

---

## Context

Light Speed Desktop needs a widget toolkit to render its own DE chrome: taskbar/panel, app launcher, notification area, settings dialog. Smithay handles the Wayland compositor machinery, but does not provide any UI widgets. The chrome layer must:

- Render into its own Wayland layer-shell surfaces (`zwlr-layer-shell-v1`).
- Support theming (light/dark mode, accent color).
- Support touch and pointer input.
- Be written in or bindable from Rust.
- Be maintainable by a small team.

Candidates:
1. **Iced** — pure Rust immediate-mode-inspired GUI framework
2. **GTK4** — mature C library with Rust bindings (`gtk4-rs`)
3. **Slint** — declarative Rust-friendly UI framework
4. **Custom wgpu rendering** — DIY widgets on top of wgpu/Vello

---

## Decision

**Iced is the preferred toolkit** (`chrome-iced` feature, default). GTK4 is available as a compile-time alternative (`chrome-gtk4` feature) for environments where Iced is impractical (e.g., if Iced's layer-shell integration proves insufficient).

The `ChromeManager` trait abstracts the chrome layer so the rest of LSD does not care which toolkit is active.

---

## Rationale

### Iced (preferred)

| Criterion | Iced | GTK4 | Slint | Custom |
|---|---|---|---|---|
| Language | Pure Rust | Rust bindings over C | Rust + .slint DSL | Pure Rust |
| No `unsafe` in our code | ✓ | Bindings use `unsafe` internally | ✓ | ✓ |
| Layer-shell support | In progress / via `iced_layershell` | Via `gtk4-layer-shell` C extension | Limited | Full control |
| Wayland-native | ✓ (uses `winit` or `raw-wayland`) | ✓ | ✓ | ✓ |
| Touch support | ✓ | ✓ | ✓ | Manual |
| Active maintenance | Active | Very active | Active | N/A |
| Compositing overhead | Low (wgpu renderer) | Moderate (GTK's own compositor) | Low | Minimal |
| AppThere ecosystem fit | High (aligns with wgpu strategy) | Medium | Medium | High |

Iced is preferred because:
- It aligns with the AppThere ecosystem's Rust-first, wgpu-based rendering strategy.
- `iced_layershell` (community crate) enables Wayland layer-shell surfaces, which is necessary for panel and launcher chrome.
- Its wgpu renderer shares GPU context setup patterns with the rest of the AppThere codebase.
- Pure Rust, no C library dependency for the core toolkit.

### GTK4 as a fallback

GTK4 is battle-tested, has excellent accessibility support (AT-SPI), and its `gtk4-layer-shell` crate provides layer-shell surfaces. It is the safer fallback if Iced's layer-shell integration proves unstable. The `ChromeManager` abstraction ensures the WM and compositor code is not coupled to either toolkit.

### Why not Slint

Slint is well-regarded but its `.slint` declarative DSL adds a build-time compilation step and a less familiar abstraction layer. For a small team, keeping everything in plain Rust is preferable. Slint may be reconsidered if GTK4 and Iced both prove inadequate.

### Why not custom wgpu rendering

DIY widgets would provide the most control and lowest overhead, but building an accessible, themeable, input-aware widget system from scratch is equivalent to writing a UI framework — exactly the NIH syndrome the project is designed to avoid. Reserved as an escape hatch only.

---

## Consequences

- Both `iced` and `gtk4` are optional dependencies behind feature flags. The default build uses Iced.
- The `ChromeManager` trait must expose a surface lifecycle API (`create_panel`, `create_launcher`, `hide_launcher`) that both implementations satisfy.
- Iced's layer-shell integration depends on `iced_layershell` (community crate). This crate must be evaluated for stability before M9. If it is insufficient, switch to GTK4 for M9 without changing the WM or compositor code.
- GTK4 requires `libgtk-4` to be installed in the Termux environment (`pkg install gtk4`). This is an additional runtime dependency that Iced avoids.
- Accessibility: GTK4 provides AT-SPI accessibility out of the box. Iced's accessibility story is evolving; this is an accepted trade-off for the initial release.
