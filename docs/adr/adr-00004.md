# ADR-004: Android Rendering — wgpu via SurfaceView + JNI

**Status:** Accepted  
**Date:** 2025-05-03  
**Deciders:** Kevin (project lead)

---

## Context

The Waymux Client receives decoded BGRA8 frame buffers and must display them on Android as rapidly as possible, matching the compositor's frame rate. Candidates:

1. **Android Canvas / Bitmap** — draw decoded bytes into a `Bitmap`, render via `Canvas.drawBitmap` on a `SurfaceView`.
2. **wgpu (Vulkan) via JNI** — Rust wgpu renders a fullscreen textured quad onto an `ANativeWindow`.
3. **OpenGL ES via JNI** — GLES 3.0 from Rust via raw bindings or a GL crate.
4. **MediaCodec Surface** — use `MediaCodec` as a hardware decoder consuming the H.264 stream, writing directly to a `Surface`.

---

## Decision

Use **wgpu (Vulkan backend) via Rust JNI**, rendering decoded frames as a 2D texture onto an `ANativeWindow`-backed `SurfaceView`.

---

## Rationale

### Why wgpu

- **Consistency with the rest of AppThere:** Loki, Iris, and other AppThere projects already use wgpu/Vello as the rendering stack. Using wgpu here means shared knowledge and tooling.
- **Performance:** wgpu on Vulkan can upload a 1080p texture and blit it in well under 1 ms on modern Android hardware. `Canvas.drawBitmap` goes through a software path that is significantly slower for large buffers.
- **No GLES boilerplate:** wgpu abstracts away EGL context creation, surface management, and shader compilation. Writing raw GLES from Rust requires either `glutin` (desktop-only) or manual EGL bindings.
- **Future-proof:** wgpu supports compute shaders, enabling future in-client post-processing (scaling, color correction) without changing the rendering architecture.

### Why SurfaceView (not TextureView)

- `SurfaceView` renders into a separate layer managed by `SurfaceFlinger`, bypassing the Android View hierarchy. This eliminates compositing overhead and reduces frame latency.
- `SurfaceView` is the correct choice when the rendered content is the primary content of the screen and does not need to be animated or transformed within the View hierarchy.
- `TextureView` would add one extra compositing step and up to one frame of additional latency — undesirable for an interactive remote desktop.

### Why not MediaCodec Surface

- MediaCodec with H.264 decoding is the lowest-latency path **when the source is an H.264 bitstream**, because it uses the hardware video decoder and writes directly to the display pipeline.
- However, Waymux Bridge's primary encoding tier is zstd-compressed BGRA8, not H.264. Decoding that on Android requires software decompression followed by a texture upload — which is exactly what wgpu provides efficiently.
- H.264 support is reserved for a future tier (ADR-002). When implemented, MediaCodec Surface becomes the preferred path for that tier, and `waymux-client-rs` can conditionally use it when the bridge negotiates H.264.

### JNI boundary design

- The Kotlin `Activity` manages the Android lifecycle and obtains the `Surface` object from a `SurfaceHolder`.
- The `Surface` is passed to Rust via JNI; Rust calls `ANativeWindow_fromSurface` (one `unsafe` block, documented with a `// SAFETY:` comment).
- All subsequent rendering happens in Rust on a dedicated tokio blocking thread, never blocking the Android main thread.
- The JNI boundary is kept minimal: surface lifecycle events and opaque input byte arrays only. No rich Kotlin types cross the boundary.

---

## Consequences

- The `waymux-client-rs` crate has one `unsafe` block (at the `ANativeWindow_fromSurface` call). This is the sole exception to the no-`unsafe` rule, and it is explicitly justified by FFI necessity.
- Cross-compilation requires the Android NDK and `cargo-ndk`. The CI pipeline must install the NDK and configure the `aarch64-linux-android` target.
- `wgpu` must be compiled with the `vulkan` feature. The Vulkan Portability layer (MoltenVK / SwiftShader) is not needed for Android, which has native Vulkan support from Android 7+.
- The `tracing-android` crate routes Rust log output to Android `logcat`, making debugging from `adb logcat` straightforward.
