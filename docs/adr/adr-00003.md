# ADR-003: Compositor Library — Smithay

**Status:** Accepted  
**Date:** 2025-05-03  
**Deciders:** Kevin (project lead)

---

## Context

Light Speed Desktop requires a Wayland compositor library that handles Wayland protocol machinery so the project can focus on window management policy, input handling, and DE chrome. Candidates:

1. **Smithay** — pure Rust Wayland compositor framework
2. **wlroots** (via `wlroots-rs` bindings) — C library with Rust FFI bindings
3. **Build from scratch** using `wayland-server` crate directly

Additionally, the Waymux Bridge requires a Wayland **client** library to connect to an existing compositor. Candidates:

1. **smithay-client-toolkit (SCTK)** — Rust, built on `wayland-client`
2. **wayland-client** directly — lower level, more manual work

---

## Decision

- **Light Speed Desktop compositor:** Use **Smithay** (`smithay` crate, version 0.7+).
- **Waymux Bridge Wayland client:** Use **smithay-client-toolkit** (`smithay-client-toolkit` crate, version 0.19+) built on `wayland-client`.

---

## Rationale

### Smithay for the compositor

| Criterion | Smithay | wlroots-rs | Raw wayland-server |
|---|---|---|---|
| Language | Pure Rust | Rust FFI over C | Pure Rust |
| Safety | Rust ownership model | `unsafe` at every wlroots call | Manageable |
| Maintenance | Active (System76 COSMIC, others) | Sparse, wlroots API churn | N/A |
| Protocol coverage | Core + xdg-shell + wlr-protocols + KDE | Core + wlr-protocols | Manual per-protocol |
| XWayland | Built-in helper | Via wlroots | Manual |
| DRM/KMS backend | Built-in | Via wlroots | Manual |
| Screencopy extension | Built-in | Via wlroots | Manual |
| No `unsafe` crate policy | Compatible | **Incompatible** — wlroots is C | Manageable |

Smithay is the clear choice:
- It is the only production-grade, actively maintained pure-Rust compositor library.
- It is used by System76's COSMIC desktop and several other projects, demonstrating real-world stability.
- It avoids the `unsafe` block proliferation that wlroots-rs would require.
- It provides built-in helpers for DRM/KMS, XWayland, screencopy, and virtual keyboard/pointer — all required features.

The project's "`no unsafe` except at FFI boundaries" policy rules out wlroots-rs as a primary dependency.

### SCTK for the bridge client

SCTK is the standard Rust high-level Wayland client library maintained by the Smithay organization. It handles global enumeration, surface lifecycle, shm buffer management, and seat handling — everything the bridge needs as a client. Using `wayland-client` directly would require reimplementing this infrastructure manually.

---

## Consequences

- Smithay uses `calloop` as its event loop. Tokio is still used for the async Unix socket server in the bridge, bridged to calloop via channels.
- Smithay's `unsafe` surface internally (for raw Wayland protocol bindings) is encapsulated behind its API — our code remains `unsafe`-free.
- If Smithay's API changes between 0.7.x minor versions, the project must track updates. Pin the exact minor version in `Cargo.toml` and audit changes before updating.
- The `wayland-protocols-wlr` crate (for wlr-screencopy, wlr-virtual-pointer) is maintained alongside Smithay and is the authoritative Rust binding for wlr-protocol extensions.
